# Multiplayer Session Lifecycle
This document outlines the end-to-end lifecycle of a collaborative session in Cognitive Resonance from the user's perspective. It describes how users connect, interact with both humans and AI, and handle offline scenarios within the Local-First Event-Sourced architecture.

## 1. Genesis: Session Creation & Invites
Every multiplayer interaction begins with a host user creating a session.

1. **Initialization:** User A (Host) starts a new session via `/session new` in the CLI or PWA. The session is immediately available in their Local-First database.
2. **Generating the Link:** User A decides to bring a colleague on board. They generate an invite link (e.g., via the PWA "Invite" button or `/invite` CLI command). Under the hood, this signals the Cloudflare backend to mint a unique, short-lived sharing token granting access to the underlying `sessionId`.
3. **Joining:** User B (Guest) clicks the link. Their client connects directly to the Cloudflare Durable Object managing that specific room, simultaneously fetching the historical Event stream from D1 to materialize the chat history, artifacts, and semantic graph locally.
4. **Read-Only vs. Co-Op:** By default, invited guests join in an active cooperative state. (If User A specifically generated a read-only link, User B can only view the timeline and must execute a `/session clone` to continue independently).

## 2. Live Interaction: The "Cocktail Party"
Once connected, users enter a high-speed, synchronized workspace.

1. **Human-to-Human Chat:** User A and User B chat normally in the text field. The UI distinguishes messages visually based on the `actor` field in the Event payload. These messages bounce sub-millisecond across the Cloudflare WebSocket (Durable Object) and update both screens instantly.
2. **Silent AI:** While humans chat, the AI (e.g., `gemini-1.5-pro`) is a "silent observer". It consumes the streaming context into its sliding window but *does not generate output*, preventing unwanted interruption.
3. **Explicit Awakenings (`@`):** User A wants the AI's input on a debate. They type `@SystemCoder, what is the best architecture here?`
4. **Multi-Agent Evaluation:** The system detects the explicit trigger. The backend packages the recent multi-user context—annotating exactly who said what—and dispatches it to the LLM. The AI responds within the thread, addressing both users.
5. **Artifact Sync:** User B opens a markdown artifact (e.g., `architecture.md`) within the session. Utilizing the Git Object Protocol, their edits form branching commits that are pushed. User A's UI indicates an update is available or auto-merges the payload.

## 3. The Edge Case: Offline & Disconnection
The Local-First topology ensures the application never completely breaks when the network goes down.

1. **Going Dark:** User B gets on an airplane and loses Wi-Fi. Their WebSocket drops. 
2. **Local Event Queueing:** User B continues working. They modify an artifact and type out a long prompt. Because the UI is bound to the local SQLite/IndexedDB, these actions are logged normally as immutable Local Events. The AI generation will fail (requiring connection to Gemini API), but the prompt is saved.
3. **Asynchronous Divergence:** Meanwhile, User A continues chatting with the AI online. Both users' timelines are now diverging.

## 4. Reconciliation: The Reconnection Merge
When the network is restored, the system must harmonize the competing timelines.

1. **The Sync Daemon:** Upon reconnecting to Wi-Fi, User B's background `syncDaemon` fires immediately.
2. **Event Push:** User B's client pushes its queued, un-synced events (the artifact edit and the offline prompt) to the Cloudflare Edge `/api/events/batch` endpoint.
3. **Event Pull & CRDT:** Simultaneously, User B's client pulls down the events generated by User A and the AI while offline.
4. **Chronological Weaving:** Because every Event is stamped with an immutable global timestamp and previous parent ID, the local UI rebuilds the Chat List dynamically. User B's offline prompt is slotted into the timeline accurately, and User A's AI interactions appear smoothly after.
5. **Artifact Merge:** If User A also edited `architecture.md` while User B was offline, the underlying `isomorphic-git` subsystem detects a merge conflict upon the remote `git pull` attempt and prompts the users to resolve the file state.

## 5. Branching & Conversion (Single-User Mode)
While working in a multiplayer session, users might want to explore an idea without forcing the rest of the room down a specific rabbit hole.

1. **Session Forking (Cloning):** At any time, any connected user (or guest viewer) can execute a "Clone" or "Fork" operation on the active session.
   - This creates an exact independent duplicate of the session up to the *current turn*.
   - The newly cloned session is completely independent from the original Durable Object room; changes made in the clone will not broadcast to the original multiplayer room.
2. **Converting to Single-User:** When a multiplayer session is cloned, the resulting duplicate is inherently a **single-user session** owned purely by the user who executed the clone. 
   - They become the sole host of that local timeline.
   - If they wish to re-invite people later, they would have to generate a completely new invite token for that specific clone.

## 6. Sunset: Archival & Closure
1. **Closing the Room:** Eventually, users close the tab. The Durable Object drops their WebSocket connections and removes them from the "Active Peers" presence list. The room goes dormant.
2. **Archival:** Days later, User A (the Host) decides the objective is complete. They execute an `archive` action. The session is moved into a read-only historic state locally, and any invite tokens associated with it are invalidated.
3. **Edge Persistence:** Regardless of archival, all events remain durably stored in Cloudflare D1, and the aggregated conversation semantics reside permanently indexed within Cloudflare Vectorize for future cross-session `/graph search` operations.
