# User IDs and Sessions

Users are identified by their unique global IDs. 

Within a given session, users are identified by their session IDs (nicks), which are unique within the session. 

Users may have nicks. If nicks are not provided, they are generated by the system deriving them from the user's global ID (or email).

The system resolves nicks name conflicts automatically by appending a number to plain nicks. If a user with nick "Bob" joins the session, and there is already a user with the same nick, the new user will be assigned the nick "Bob2", and even if the original "Bob" leaves the session before the other Bob joins, the new user will keep the nick "Bob2".

In the session records all the user and their AI's actions are associated with the user's global ID. This allows to track user activity across sessions. User nicks are recorded as well for easy session review and resuming.

## The Session Lifecycle Model

To ensure a robust, predictable user experience, session management follows a strict lifecycle model centering on explicit user intent. Complex multi-dimensional merging of sessions is intentionally omitted to keep the mental model obvious and intuitive.

### Core Operations

1. **Creation:** A user starts a completely new, blank session.
2. **Duplication (Forking/Cloning):** A user creates an independent, exact copy of an existing session (whether active or archived). This is used to explore an alternative tangent without polluting the original timeline. The Duplicate is computationally an entirely new session record with its own unique identifier.
3. **Archival:** A user explicitly moves an active session into an "Archived" state. This hides it from their primary active views. Archived sessions act safely as read-only historical records to prevent accidental tampering.
4. **Retrieval (Unarchiving):** A user restores an archived session back to their active view, making it fully read-write and active again.
5. **Permanent Deletion:** An irreversible destructive action where the session record is completely eradicated from the storage medium. For safety, it is highly recommended that sessions be in an Archived state before they are presented as eligible for permanent deletion, acting as a two-step trash can process.

### User Interface (UI) Requirements

To support these lifecycle operations while protecting the user from accidents, the UI must implement two primary safety mechanisms:

- **Undo Bubbles (Toast Notifications):** For non-destructive state changes (like Archival or Retrieval), the UI should immediately perform the action on click but present an ephemeral "Bubble" (e.g., "Session Archived. [Undo]"). This avoids friction by removing blocking modals while maintaining safety.
- **Confirmation Prompts (Modals):** For inherently destructive and irreversible actions (like Permanent Deletion), the UI must present an explicit, blocking confirmation modal outlining the consequences before executing the deletion to the backend.

### Command Line Interface (CLI) Requirements

The CLI must mirror the safety and flexibility of the UI, optimized for both human operators and automated scripts:

- **Interactive Confirmation:** Commands that perform destructive actions (like `session delete <id>`) must halt execution and prompt the human interactively (e.g., `Are you sure you want to permanently delete this session? [y/N]`).
- **Force Flags:** To support automation and power users, commands that normally require interactive confirmation must accept a `--force` or `-f` flag to bypass the safety prompt gracefully (e.g., `session delete <id> --force`).

### Read-Only Access and Collaboration

When users want to share their work sequentially without engaging in complex live-multiplayer collisions, they utilize the Read-Only workflow:

1. **Sharing for Review:** User A grants User B read-only access to a specific session (e.g., via a unique URL or share link). User B can view the entire chronological history, artifacts, and context, but their interface strictly disables input, generation, or modification of the source session.
2. **Cloning to Continue Development:** If User B wants to build upon User A's work, User B can explicitly execute a **Clone (Duplicate)** operation on the read-only session. This duplicates the session data, assigning User B as the owner of the newly created, independent copy. User B can now confidently continue development and exploration without affecting User A's original work.
